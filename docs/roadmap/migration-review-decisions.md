# Migration Review Decisions

**Created:** 2026-05-08
**Purpose:** Shared decision register for recurring migration findings.

Use this file when the same pattern appears across multiple agents, skills, guides, or templates. The migration workflow is:

1. create a copy-as-is baseline,
2. run similarity and advisory universality scans,
3. record recurring decisions here,
4. apply those decisions consistently in later files.

---

| MRD-0022 | QA runtime URLs and shared test-account documents | Render from config and create or reuse during init | Config | QA agents, artifact contracts, init | QA is core, but a local URL such as `localhost:8787` and a shared-account doc path vary by project. | MRD-0024 | Worktree and GitHub delivery workflow guidance | Keep GitHub PR delivery semantics in core; render branch names, remote refs, repository cleanup commands, task prefixes, and helper-script names from config or installed Agent Flow scripts | Core + Config | Delivery guides, delivery agents, init, scripts | GitHub PR flow is common enough to be core, but project branch policy, remotes, repository refs, local paths, and helper availability vary by install. Core guides should explain the flow while `agent-flow config explain` and installed scripts make concrete commands transparent. | Accepted |
| MRD-0025 | External UI/design reference guides such as `ui-ux-pro-max` | Migrate with the owning design pack, not as standalone core | Pack + Vendor | Design guides, UI skills, nested skill assets | The content is useful for Agent Flow installs, but it is vendor-derived and tied to a related skill/data/search-script bundle. It should be installed only when the `design` pack is selected, with provenance noted in the migration record. | Accepted |
| MRD-0026 | Pack-owned static skill assets with docs, data, scripts, JSON, or license files | Render as pack skill assets with target path adaptation and file-type-compatible managed metadata | Pack + Target wrapper | Skills, nested skill assets, renderer, validation | Skills are part of Agent Flow installs, but optional skill bundles should stay pack-owned. Markdown, scripts, and CSV files can carry managed metadata in comments; JSON and license text remain raw to avoid breaking parsers or altering license content. Scripts that read managed CSV files should skip the generated metadata comment. | Accepted |
| MRD-0027 | Static skills referenced directly by core agents | Render as core static skills even when provenance is vendor/adapted | Core + Vendor | Core agents, skills, renderer | Core agent instructions must be self-contained in a zero-pack install. If a core agent links to a skill, that skill cannot require an optional pack just to avoid a broken local link. Vendor provenance remains recorded separately from the core-vs-pack render decision. | Accepted |
| MRD-0028 | Static skill assets need project values such as checks, branch names, artifact paths, or target-specific agent names | Render known Agent Flow placeholders in static skill assets; leave unrelated mustache examples literal | Config + Target wrapper | Skills, renderer, validation | Some copied skills are mostly static but still need portable values. Static rendering should support known contexts like `git.*`, `checks.*`, `artifacts.*`, and `target.*` without treating every `{{example}}` snippet in vendor docs as a required template variable. | Accepted |
| MRD-0029 | Repo-local delivery skills whose source content hardcodes one project's branch/release policy or command set | Universalize through config; document any structural release-policy exception below the 99% aspirational target | Config | Commit and release skills | Branch names, release branches, status docs, helper scripts, npm commands, and co-author trailers are installer/project-specific. Straight substitutions should be normalized in the similarity gate. Larger release-policy rewrites may fall below 99%, but only when the file-specific migration status explains why. | Accepted |
| MRD-0030 | Core skills or aliases that point to another Agent Flow skill | Migrate the referenced skill in the same or immediately following batch | Core | Skills, renderer, validation | Zero-pack core installs must not render self-contained skills that send the user to a missing local skill. Alias files may stay minimal, but their canonical target must render wherever the alias renders. | Accepted |
