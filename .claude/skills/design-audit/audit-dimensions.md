# Audit Dimensions — Reference Guide

8 измерений UX-аудита с severity criteria. Дополняет 80-item checklist из `SKILL.md`.
Адаптировано под документационную платформу (редактор, published docs, AI-ассистент).

---

## Когда какой аудит

**Full audit (все 8 dimensions):**
- Новая страница / новый экран
- Redesign существующего UI
- Feature с layout-изменениями
- Низкая adoption после релиза

**Targeted audit (2-3 dimensions):**
- Правка одного компонента (кнопка, модалка, тултип)
- Багфикс UI
- Конкретная жалоба ("не могу найти X" → IA + Context)

Всегда аудитируй против **реальных задач пользователя**, не дизайнерского замысла.
Вопрос не "красиво ли?", а "может ли пользователь выполнить задачу эффективно?"

---

## 1. Information Architecture

**Что это:** Структура, лейблы, группировка контента.

**Что искать:**
- Primary action/content показан первым или закопан?
- Связанные элементы сгруппированы визуально и семантически?
- Несвязанные элементы случайно сгруппированы (proximity создаёт ложную связь)?
- Лейблы используют термины пользователя или разработчика?
- Навигация sidebar: порядок соответствует частоте использования?

**Severity:**
- **Critical:** Primary task требует поиска по нескольким секциям или недоступен
- **Major:** Связанные данные разнесены по несвязанным областям
- **Minor:** Лейблы с жаргоном; группировка неточная, но понятная
- **Enhancement:** Переупорядочивание улучшило бы scan path

**Типичные findings для ZNAI:**
- "Создать документ" закопан за 3+ клика с dashboard
- Настройки пространства и настройки документа в одном разделе
- AI-ассистент отделён от контекста документа, который он анализирует
- Sidebar: техническая структура (по API) вместо ментальной модели пользователя

## 2. Visual Hierarchy

**Что это:** Соответствие визуального веса элементов их информационной важности.

**Что искать:**
- На что глаз падает первым? Это правильная вещь?
- Primary content визуально крупнее/жирнее secondary?
- Декоративные элементы (бордеры, фоны, иконки) конкурируют с контентом?
- Typography дифференцирована достаточно (размер, вес)?
- Error/warning states визуально выделяются?
- В editor: toolbar не перетягивает внимание от контента?

**Severity:**
- **Critical:** Опасное состояние (unsaved changes, sync conflict) визуально неотличимо
- **Major:** Самый важный элемент имеет тот же визуальный вес что и secondary
- **Minor:** Иерархия есть, но subtle — пользователь замедляется
- **Enhancement:** Усиление контраста или размера ускорило бы сканирование

**Типичные findings:**
- Все пункты sidebar одного веса — активный документ неотличим
- Кнопка публикации стилизована так же, как второстепенные действия
- AI-ответ визуально неотличим от пользовательского ввода

---

## 3. Screen Real Estate

**Что это:** Используется ли пространство пропорционально важности контента.

**Что искать:**
- Большие пустые области рядом с плотными, трудночитаемыми?
- Редко используемая фича занимает фиксированный крупный блок?
- Padding избыточный за счёт контент-плотности?
- Что-то вызывает горизонтальный скролл?
- Collapsible/tabbed секции скрывают то, что нужно при каждом посещении?

**Severity:**
- **Critical:** Ключевой контент за interaction (tab/scroll) на каждом page load для primary workflow
- **Major:** Основной контент требует скролла мимо крупной декоративной/второстепенной секции
- **Minor:** Padding щедрый, но контент не скрыт
- **Enhancement:** Уплотнение позволило бы видеть больше без скролла

**Типичные findings:**
- Sidebar занимает 30%+ ширины при 3 пунктах навигации
- Editor toolbar дублирует slash-commands, занимая вертикальное пространство
- Empty state иллюстрация на весь экран вместо компактного CTA
- Published docs: оглавление скрыто за бургером на desktop

## 4. Interaction Cost

**Что это:** Количество шагов (клики, скроллы, переходы) до ключевой информации или действия.

**Что искать:**
- Сколько кликов до primary workflow?
- Нужно ли покидать текущую страницу для контекста?
- Частые действия на виду или закопаны в меню?
- Форма требует заполнения полей, которые можно auto-populate?
- Confirmation dialogs для low-risk действий?

**Severity:**
- **Critical:** Primary task = 5+ шагов, когда 1-2 реально
- **Major:** Частый context-switching (проверить значение на странице A, заполнить форму на B)
- **Minor:** Действие требует 1 лишний клик
- **Enhancement:** Keyboard shortcuts ускорили бы power users

**Типичные findings:**
- Создание документа: dashboard → пространство → "+" → модалка → заполнение → создать (5 шагов)
- Просмотр AI-ответа и исходного документа требует переключения вкладок
- Приглашение участника: settings → members → invite → email → role → send
- Поиск по документации не сохраняет query при навигации назад

## 5. Cognitive Load

**Что это:** Ментальное усилие для понимания и действия на странице.

**Что искать:**
- Больше 7 +/- 2 элементов в списке/меню без группировки?
- Все данные flat, без progressive disclosure?
- Нужно запоминать информацию из одной секции для другой?
- Есть unlabeled иконки, требующие обучения?
- Editor toolbar: все команды разом или по категориям?

**Severity:**
- **Critical:** Страница содержит противоречивые или ambiguous элементы
- **Major:** Пользователь должен держать несколько pieces в working memory одновременно
- **Minor:** Секция плотная, но парсится с усилием
- **Enhancement:** Progressive disclosure или inline guidance сгладили бы обучение

**Типичные findings:**
- Editor toolbar с 20+ кнопками без группировки
- Settings страница: все настройки flat без категорий
- AI-ассистент показывает raw source links без preview
- Permissions: роли, пространства и документы — три уровня без визуальной связи

## 6. Context & Orientation

**Что это:** Знает ли пользователь где он, какие фильтры активны, какое состояние системы.

**Что искать:**
- Текущий документ/пространство всегда видны?
- Активные фильтры отображаются persistently?
- Текущая страница/секция выделена в навигации?
- После действия (save, publish) — UI подтверждает что произошло?
- При stale/cached данных — это коммуницируется?
- Breadcrumb или back-path на drill-down страницах?

**Severity:**
- **Critical:** Пользователь не может определить какой документ/пространство просматривает
- **Major:** Активные фильтры невидимы; пользователь не понимает что данные отфильтрованы
- **Minor:** Breadcrumb есть, но неточный или неполный
- **Enhancement:** Timestamp последнего обновления повысил бы доверие

**Типичные findings:**
- Yjs sync status невидим — пользователь не знает, сохранены ли изменения
- Published site: нет индикации какая версия опубликована
- После publish нет persistent confirmation — только toast на 3 секунды
- AI chat: нет индикации какие документы используются как контекст

## 7. Data Presentation

**Что это:** Показаны ли данные в правильном формате и с правильной точностью.

**Что искать:**
- Сравнения (версии документов, статистика) очевидны?
- Даты форматированы консистентно и в locale пользователя?
- Empty/null значения: blank, zero, "---"? Консистентно?
- AI confidence/relevance показаны понятно?
- Длинные списки документов: пагинация или infinite scroll?

**Severity:**
- **Critical:** Формат искажает данные (неправильный порядок версий, обрезанный контент)
- **Major:** Данные без контекста, требующие ментальных вычислений
- **Minor:** Inconsistent форматирование
- **Enhancement:** Добавление индикатора (relative date, delta) ускорило бы понимание

**Типичные findings:**
- AI search results: relevance score в процентах без объяснения
- Document history: абсолютные даты вместо relative ("3 дня назад")
- Analytics: "0 просмотров" неотличимо от "нет данных"

## 8. Responsiveness & Edge Cases

**Что это:** Поведение при граничных условиях — empty data, loading, errors, mobile, длинный текст.

**Что искать:**
- Meaningful empty state (первый запуск, нет результатов, нет данных)?
- Loading states информативные?
- Error states actionable (пользователь знает что делать)?
- Layout не ломается на mobile/narrow viewports?
- Что происходит с очень длинными названиями документов?
- 0, 1, 2, 200+ документов в пространстве — все случаи?
- Hardcoded heights, обрезающие контент?
- Yjs offline/reconnection — что видит пользователь?

**Severity:**
- **Critical:** Error state = blank screen или raw stack trace
- **Major:** Empty state primary list = ничего — пользователь не знает: данных нет или баг?
- **Minor:** Длинный текст обрезан без tooltip — информация потеряна
- **Enhancement:** Empty state с CTA ("Создать первый документ")

**Типичные findings:**
- Пустое пространство: белый экран без guidance
- AI chat: spinner крутится бесконечно при API error — нет timeout
- Sidebar: длинные названия документов обрезаны без tooltip
- Published site: 404 без навигации назад к другим страницам

---

## Severity Interaction Rule

> Когда несколько findings комбинируются (например: buried data + no orientation = пользователь не может найти И не знает на что смотрит), escalate combined severity на один уровень.

Пример: IA Minor + Context Minor = комбинированный **Major**, потому что пользователь одновременно не находит нужное и не понимает где он.

---

## Связь с 80-Item Checklist

Dimensions и checklist — комплементарны:
- **Checklist** (SKILL.md) = конкретные визуальные/технические пункты. "WCAG AA 4.5:1 contrast?" — pass/fail.
- **Dimensions** (этот файл) = task-oriented framework. "Может ли пользователь создать документ за ≤2 клика?" — с severity criteria.

При аудите: checklist для механической проверки, dimensions для user-task-oriented анализа.
