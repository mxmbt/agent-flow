# UX Principles — Reference Guide

Теоретическая база для обоснования UX findings.
Адаптировано под документационную платформу (редактор, published docs, AI-ассистент).

---

## 1. Паттерны сканирования

### F-Pattern

Срабатывает на текстовых/табличных layouts (editor, документация, настройки):
- Пользователь сканирует верхнюю горизонтальную полосу полностью
- Опускается, сканирует вторую короткую горизонтальную полосу
- Сканирует левый край вертикально

**Импликация:** критическая информация — top-left. Sidebar навигация — слева. Правая сторона часто пропускается.

**Для ZNAI:** В editor контент = центр внимания. Sidebar (навигация по документам) — слева, toolbar — сверху. Действия справа (publish, share) — менее заметны, требуют визуального усиления.

### Z-Pattern

Срабатывает на sparse/visual-first layouts (landing, empty states, onboarding):
- Глаз: лево-право сверху → диагональ вниз → лево-право снизу

**Для ZNAI:** Published docs landing, пустые пространства, onboarding screens.

### Правило 3 секунд

Новый пользователь должен определить 3 самых важных элемента на странице за 3 секунды. Если нет — визуальная иерархия сломана.

## 2. Visual Hierarchy — 5 рычагов

| Рычаг | Как использовать | Типичная ошибка |
|-------|-----------------|-----------------|
| **Размер** | Крупнее = важнее. Заголовки 32-48px vs body 14-16px | Всё крупное — иерархия разрушена |
| **Цвет** | Один accent для critical/actionable. Red для alerts | 6+ цветов — пользователь не строит ментальную модель |
| **Контраст** | Высокий = foreground. Низкий = secondary | Светло-серый на белом для нужного контента |
| **Spacing** | Whitespace группирует связанное, разделяет несвязанное | Плотная укладка — всё равноважно |
| **Типографика** | Bold для значений, regular для лейблов | Болдинг лейблов вместо значений |

## 3. Information Scent

Способность пользователя предсказать, что произойдёт при клике.

**Сильный scent:**
- Лейблы соответствуют ментальной модели ("Документы", не "Ресурсы")
- Числовые preview ("Приглашения (3)", не просто "Приглашения")
- Консистентная иконографика
- Breadcrumbs показывают где ты, не только откуда пришёл

**Слабый scent:**
- Generic лейблы ("Подробнее", "Детали", "Открыть")
- Unlabeled иконки (допустимы только для universal: home, search, close)
- Collapsed секции без preview содержимого

**Тест:** спросить пользователя, который никогда не видел страницу: "что будет, если нажать X?" Если не может уверенно ответить — scent слабый.

## 4. Progressive Disclosure

Показывать только нужное для текущей задачи, раскрывая сложность по запросу.

### Три уровня для ZNAI

| Tier | Видимость | Примеры |
|------|-----------|---------|
| **Tier 1** — всегда видно | 0 interactions | Название документа, статус публикации, список страниц, основные действия (edit, publish) |
| **Tier 2** — 1 клик | Expand, tooltip, drawer | Версия документа, AI-suggestion detail, настройки отдельной страницы, member permissions |
| **Tier 3** — навигация | Отдельная страница | Полная история изменений, analytics, audit log, bulk настройки пространства |

**Типичная ошибка:** Progressive disclosure для данных, которые нужны постоянно. Если пользователь каждый раз разворачивает одну и ту же секцию — это Tier 1, не Tier 2.

## 5. Cognitive Load

### Miller's Law (7 +/- 2)

Working memory = ~7 chunks. Chunk — смысловая единица, не item.

**Для ZNAI:**
- Sidebar: группировать документы по секциям (chunks), а не flat список
- Editor toolbar: ≤7 видимых групп инструментов
- Settings: категоризировать, не показывать все 20 настроек разом
- AI results: группировать по source/relevance, не flat list

### Hick's Law

Время решения растёт логарифмически с количеством вариантов.

**Для ZNAI:**
- "Создать документ" → умные defaults (текущее пространство, формат по умолчанию)
- Slash-commands: показывать top-5 по частоте, остальные через search
- Tone of Voice: preset "Профессиональный" по умолчанию, кастомизация по запросу

### Decision Fatigue

При длительной работе качество решений деградирует.

**Для ZNAI:**
- Auto-save вместо "Сохранить?"
- AI-drafts: "Принять / Отклонить" вместо "Выберите из 5 вариантов"
- Bulk operations для повторяющихся действий (publish all, move multiple)

## 6. Gestalt Principles

### Proximity
Элементы рядом воспринимаются как связанные. Consistent spacing для группировки.

**Для ZNAI:** В editor — spacing между блоками сигнализирует связь/разрыв. В sidebar — группы документов.

### Similarity
Одинаково выглядящие элементы предполагают одинаковое поведение.

**Для ZNAI:** Все кликабельные карточки документов должны выглядеть одинаково. Смешивание interactive и non-interactive с идентичным стилем = confusion.

### Continuity
Глаз следует линиям и кривым. Alignment создаёт visual flow.

**Для ZNAI:** Выравнивание sidebar items, editor blocks, settings rows по одной оси.

### Closure
Неполные формы воспринимаются как целые.

**Для ZNAI:** Progress bars (публикация, AI processing), circular indicators (completion %).

### Figure-Ground
Пользователь разделяет foreground (контент) и background (контейнер).

**Для ZNAI:** Dark mode: проверить все цвета на тёмном фоне. Editor content должен быть визуально "выше" окружения.

## 7. Change Blindness

Пользователь не замечает визуальные изменения, если внимание не направлено на изменяющийся элемент.

**Для ZNAI:**
- Yjs sync: если коллаборатор изменил текст — brief highlight или delta badge
- AI draft: анимация появления, не silent insert
- Статус документа изменился (draft → published) — visual confirmation, не только silent update
- Notifications: toast недостаточно — persistent badge для важных событий

**Pre-attentive attributes** (обрабатываются до сознательного внимания, ~50ms):
- **Цвет** — красный среди зелёных элементов
- **Размер** — крупный элемент среди мелких
- **Движение** — анимация в статичном UI (использовать sparingly)

Используй pre-attentive для исключений и critical states. Не для декорации.

## 8. Shneiderman's Mantra

> **Overview first, zoom and filter, then details on demand.**

### Для ZNAI — 3 слоя

**Layer 1 — Overview (0 interactions):**
- Dashboard: список пространств, количество документов, последняя активность
- Пространство: список документов, статус публикации, кто последний редактировал
- Ответ на: "Всё в порядке? Есть что-то требующее внимания?"

**Layer 2 — Zoom and filter (1-2 interactions):**
- Фильтр документов по статусу, автору, дате
- Клик на документ → preview/editor
- AI search → результаты с context snippets
- Ответ на: "Какой документ мне нужен? Что изменилось?"

**Layer 3 — Details on demand (3+ interactions):**
- Полная история версий
- Детальная аналитика просмотров
- Audit log: кто, когда, что изменил
- Raw export (Markdown, PDF)
- Ответ на: "Что конкретно изменилось в версии 3?"

## 9. Anti-Patterns для документационных платформ

### Tab Overuse
**Проблема:** Tabs фрагментируют связанные данные. Пользователь сравнивающий "Настройки" и "Участники" должен запоминать данные из Tab A смотря на Tab B.

**Когда tabs уместны:** Переключение между fundamentally different modes (Editor / Preview / Published).
**Лучше:** Scrollable page с секциями + anchor links.

### Settings Sprawl
**Проблема:** Все настройки в одном flat списке. Пользователь не находит нужную.

**Cure:** Группировка по задачам ("Публикация", "Доступ", "Внешний вид"), не по техническим категориям ("API", "Webhooks", "Tokens").

### Equal Visual Weight
**Проблема:** Все элементы одного размера/веса — пользователь читает всё, не понимая что важно.

**Cure:** Map каждый элемент на tier (critical / important / supporting). Визуальный vocabulary для каждого tier. Без исключений.

### Context Switching
**Проблема:** Для одной задачи нужно переходить между страницами. Каждый переход — перезагрузка ментальной модели.

**Для ZNAI:**
- AI-ассистент + документ = одна страница (split view), не отдельные экраны
- Настройки публикации доступны из editor, не только из settings
- Inline lookups: hover на ссылку → preview связанного документа

### Over-Reliance on Tables
**Проблема:** Таблицы — path of least resistance. Но для трендов, сравнений, пропорций — chart лучше.

**Для ZNAI:** Analytics: линейный график "просмотры по дням" лучше таблицы с 30 строками. Документ history: timeline лучше таблицы версий.

## 10. Quick Reference

### Checklist: Information Architecture
- [ ] Primary task определяется за 3 секунды?
- [ ] Самый важный элемент top-left или визуально доминирует?
- [ ] Interactive и non-interactive элементы визуально различимы?

### Checklist: Cognitive Load
- [ ] ≤7 items в любом списке/меню без группировки?
- [ ] Фильтры с умными defaults (не пустые при загрузке)?
- [ ] ≤5 top-level navigation items без группировки?

### Checklist: States
- [ ] Empty state для каждого списка (что тут будет + как добавить)?
- [ ] Loading skeleton соответствует layout контента?
- [ ] Error state включает recovery action?

### Checklist: Progressive Disclosure
- [ ] Overview отвечает на "всё в порядке?" без interaction?
- [ ] Detail доступен за ≤2 клика от overview?
